# 專案規格書：Open Data Issue Lab (開放資料議題實驗室)

## 1. 系統架構總覽 (System Architecture)

採用 **Serverless Jamstack 架構**，達到零伺服器維護成本、極速載入與高資安標準。

- **資料源 (Data Source):** data.gov.tw (台灣電力公司_過去電力供需資訊 API / CSV)。
    
- **自動化處理 (Data Pipeline):** GitHub Actions（每日定時觸發 Python 爬蟲與資料清洗）。
    
- **靜態代管 (Hosting):** GitHub Pages。
    
- **前端渲染 (Frontend):** 靜態 HTML/JS/CSS 搭配 Apache ECharts (或 Chart.js) 進行資料視覺化。

## 2. 網站層次與畫面設計 (Site Hierarchy & UI/UX)

網站分為兩大層級，確保「開放資料精神」與「具體議題」的層次分明。

- **視覺風格：** 專業、冷靜的公有雲風格（白/淺灰背景，搭配深藍色字體）。
    
- **Hero Section (主視覺區):**
    
    - 大標題：「讓數據說話：政府開放資料的社會實境秀」

- **模組導航 (Thematic Modules):** 預留未來擴充空間（如：育兒地圖、商圈存活率等），目前主推高亮顯示的 **[Case 01: 能源轉型與鴨子曲線]**。

## 3. 圖表套色與視覺化規格 (Visualization Spec)

處理能源資料，「色彩心理學」極度重要，必須符合民眾對特定能源的直覺認知，且在堆疊圖中具有高辨識度。

### 統一能源配色碼 (Color Palette)

- ☀️ **太陽光電 (Solar):** `#FFD700` (明亮的黃色 - 代表陽光，視覺上容易看出膨脹與收縮)
    
- 💨 **風力發電 (Wind):** `#2E8B57` (海綠色 - 代表自然綠能)
    
- 💧 **慣常/抽蓄水力 (Hydro):** `#1E90FF` (水藍色)
    
- ☢️ **核能發電 (Nuclear):** `#9370DB` (中紫色 - 代表高科技與特定底層基載)
    
- 🔥 **燃氣 (LNG/Gas):** `#FF8C00` (橘色 - 火力來源，近年主力)
    
- 🪨 **燃煤 (Coal):** `#696969` (深灰色 - 傳統高碳排火力)
    

### 圖表一：【微觀戰術】近 60 天日夜供電實境秀

- **圖表類型：** 可縮放堆疊面積圖 (Zoomable Stacked Area Chart)
    
- **X 軸：** 時間 (精確到小時)
    
- **Y 軸：** 發電量 (MW)
    
- **互動：** 預設顯示最近 3 天（展現最清晰的單日鴨子曲線），提供拖曳軸 (DataZoom) 讓使用者往回拉看近 60 天的極端氣候日。
    
- **圖層順序 (Z-index 概念)：** 基載（核能、煤）放在最底下，中段放燃氣，最上層放波動最大的太陽能與風力。這樣能清楚看到底層穩固，上層隨日照波動的切角。
    

### 圖表二：【宏觀戰略】近 12 個月能源結構變化

- **圖表類型：** 100% 堆疊柱狀圖 (100% Stacked Bar Chart)
    
- **X 軸：** 月份 (過去 12 個月)
    
- **Y 軸：** 發電佔比 (%)
    
- **互動：** 點擊圖例 (Legend) 可隱藏/顯示特定能源，方便單獨觀察「綠能總佔比」是否隨月份穩定上升。

## 4. 自動化資料管線 (GitHub Actions CI/CD Spec)

這是確保網站「活著」且不用付後端伺服器費用的核心。

### 💡 實務評估 (Reality Check)：GitHub Free Tier 額度

GitHub Free 帳號每個月提供 **2,000 分鐘** 的 Actions 執行時間。

我們設定每天執行一次 Python 腳本，每次抓取、處理資料到推播回 Repository，約耗時 1~2 分鐘。

**每月消耗：約 60 分鐘。** （僅佔免費額度的 3%，極度安全且綽綽有餘）。

### 工作流程設計 (Workflow Logic)

1. **觸發條件 (cron):** 設定於台灣時間每天凌晨 3:00 執行 (`cron: '0 19 * * *'` -> UTC 時間)。
    
2. **執行環境:** `ubuntu-latest`
    
3. **執行步驟:**
    
    - Checkout Repository。
        
    - 安裝 Python 環境與 `pandas`, `requests` 套件。
        
    - 執行 `data_pipeline/fetch_energy.py`。
        
    - 覆寫原本的 JSON 檔案。
        
    - 利用 Git Commit 並 Push 回 `main` 分支（這會自動觸發 GitHub Pages 重新部署靜態網站）。
        

### 輸出的資料結構 (JSON Schema)

腳本處理完後，必須吐出極度乾淨、讓前端不需做任何運算的 JSON 格式：

**`recent_60_days_hourly.json` (範例):**

JSON

```
{
  "timestamps": ["2026-05-01 14:00", "2026-05-01 15:00", "..."],
  "series": [
    { "name": "太陽光電", "data": [8500, 7200, "..."] },
    { "name": "燃氣", "data": [12000, 14000, "..."] }
  ]
}
```

## 5. 實作推動建議

按照這個 Spec，你可以將開發工作拆分為三個清晰的 Sprint（衝刺期）：

- **Sprint 1:** 寫出 Python 腳本，成功從 data.gov.tw 抓下資料並轉換為上述 JSON 格式，然後在本地端跑通。
    
- **Sprint 2:** 建立 HTML/JS 頁面，引入 ECharts 並套用設定好的「統一能源配色碼」，確保讀取本地 JSON 檔能畫出圖表。
    
- **Sprint 3:** 將程式碼推上 GitHub，撰寫 `.github/workflows/update_data.yml`，打通整條 Serverless 的血脈。
