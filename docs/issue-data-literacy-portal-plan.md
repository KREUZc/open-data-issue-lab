# 議題數字力政府開放資料網站規劃

## 1. 評估結論

現有 `open-gov-data-redesign.md` 已經適合當作 MVP：以 data.gov.tw 為來源、用 GitHub Actions 每日更新資料、以靜態網站呈現「能源鴨子曲線」議題案例。若要升級成「全部來源複刻政府資料開放平臺，但以議題數字力傳達為核心」的網站，建議不要直接複製官方網站的資訊架構，也不要在第一版完整鏡像所有原始檔案。

比較可行的定位是：

> 建立一個可追溯到 data.gov.tw 全部資料集的「來源索引」，再把高價值資料集轉化成可理解、可引用、可再製的公共議題敘事。

也就是後台「全量詮釋資料鏡像」，前台「議題導向策展」。

## 2. 產品定位

名稱可沿用現有 spec 的 **Open Data Issue Lab / 開放資料議題實驗室**，但加上一句更清楚的產品承諾：

> 從政府開放資料出發，讓一般人看懂公共議題背後的數字。

官方 data.gov.tw 的任務是資料發布、搜尋、下載與授權揭露。本專案的任務則是降低理解門檻：

- 讓使用者先從議題進入，而不是先面對資料集清單。
- 每個圖表都提供「怎麼看」、「代表什麼」、「不能代表什麼」。
- 每個數字都能回到原始資料集、提供機關、更新時間、授權條款與處理方法。
- 保留完整資料來源索引，讓進階使用者能追查、下載、重製。

## 3. 範圍分層

### Layer A：全來源索引

目標是每日同步 data.gov.tw 的資料集詮釋資料，而不是一開始就下載所有原始資料檔。

應保存欄位：

- 資料集識別碼
- 資料集名稱
- 資料集描述
- 服務分類
- 提供機關
- 檔案格式
- 資料下載網址
- 資料資源下載網址
- 更新頻率
- 詮釋資料更新時間
- 授權方式與授權說明網址
- 品質檢測狀態
- 原始頁面 URL

這一層提供「來源複刻」的可信基礎：使用者知道平台看見了哪些政府資料、資料從哪裡來、何時更新、是否可用。

### Layer B：議題資料倉

只針對已選定議題抓取、清洗、版本化原始資料。第一批議題可從現有 spec 的能源案例開始：

- 能源轉型與鴨子曲線
- 空氣品質與健康風險
- 交通事故熱點與通勤安全
- 托育、幼兒園與人口結構
- 房價、實價登錄與居住負擔

每個議題需要一份 `issue.yml` 或等價設定檔，描述：

- 議題名稱與公共問題
- 使用哪些 data.gov.tw 資料集
- 每個資料集在議題中的角色
- ETL 腳本
- 更新頻率
- 指標定義
- 圖表定義
- 已知限制

### Layer C：議題數字力前台

前台不是資料目錄網站，而是議題閱讀與互動探索工具。每個議題頁都應有固定結構：

- 一句話問題：這個議題到底在問什麼？
- 三個核心數字：用最少數字建立判讀入口。
- 互動圖表：可縮放、可切換、可比較。
- 怎麼看這張圖：避免只丟圖表給使用者。
- 資料來源卡：原始資料、提供機關、更新時間、授權、清洗方法。
- 數字限制：缺漏值、時間落差、欄位定義、不能推論的範圍。
- 下載與重製：提供處理後 JSON/CSV、原始來源連結與腳本位置。

## 4. 資訊架構

建議網站主導覽維持四個區塊：

- **議題**：能源、空氣、交通、托育、居住等主題入口。
- **資料來源**：全量 data.gov.tw 資料集索引，可搜尋、篩選、看來源健康度。
- **方法**：資料如何取得、清洗、轉換、版本化。
- **關於**：授權、免責、資料回報、專案目標。

首頁第一屏不做官方網站翻版，而要直接傳達「議題數字力」：

- 主標題：看懂公共議題背後的數字
- 副標題：所有圖表皆追溯至政府開放資料，並附上方法、限制與原始來源。
- 入口模組：今日主打議題、資料更新狀態、全站資料來源數、最新異動資料集。

## 5. 技術架構建議

### MVP 架構

沿用現有 spec：

- GitHub Actions：每日排程抓資料與產生靜態 JSON。
- Python：資料抓取、清洗、欄位正規化。
- GitHub Pages：靜態代管。
- ECharts：互動資料視覺化。
- 前端：靜態 HTML/CSS/JS 或輕量框架。

適合第一版的原因：

- 維護成本低。
- 透明，所有資料處理腳本可放在 repo。
- 很適合先完成能源案例與資料來源索引。

### 擴充架構

若要接近「全部來源複刻」，GitHub Pages 只適合放前台與小型索引，不適合承載大量原始檔鏡像。第二階段可加入：

- Object Storage：存放大型原始檔快照與處理後資料。
- SQLite/Parquet：保存資料集索引、異動歷史與品質檢測結果。
- 搜尋索引：Meilisearch、Typesense 或靜態 Pagefind，支援中文搜尋。
- Pipeline Registry：每個議題一支可重跑的資料處理管線。

建議策略是先做 metadata mirror，再做 selected resource cache，不做無差別全檔案鏡像。

## 6. 資料管線

### 每日全量索引同步

1. 下載 data.gov.tw 的全部資料集清單。
2. 轉成統一 schema。
3. 比對昨日版本，產生新增、更新、下架、下載失敗清單。
4. 輸出：
   - `public/data/catalog/datasets.json`
   - `public/data/catalog/agencies.json`
   - `public/data/catalog/categories.json`
   - `public/data/catalog/changes/latest.json`

### 議題資料同步

1. 依 `issue.yml` 找到需要的資料集。
2. 抓取原始 CSV/JSON/API。
3. 進行欄位標準化、時間格式轉換、單位轉換、缺漏值處理。
4. 輸出前端可直接使用的 JSON。
5. 產生 `methodology.json`，記錄資料來源、處理步驟與警告。

### 來源健康度檢查

每筆資料資源應標示：

- 最近成功下載時間
- HTTP 狀態
- 檔案大小
- 編碼
- 格式是否符合宣告
- 欄位是否和上次一致
- 是否需要人工檢查

這會讓網站不只是「漂亮呈現」，而是對資料品質本身有公共說明力。

## 7. 首波 MVP

建議第一版不要超過三個月標的，範圍如下：

- 首頁：議題入口、資料來源總覽、更新狀態。
- 資料來源索引：同步全部資料集詮釋資料，可搜尋與篩選。
- 能源議題頁：完成現有 spec 中近 60 天堆疊面積圖與近 12 個月能源結構圖。
- 方法頁：說明資料來源、每日更新、授權、清洗方法。
- 每張圖表的來源卡與限制說明。

MVP 成功標準：

- 使用者能在 30 秒內理解一個公共議題的核心數字。
- 每個數字都能追溯到原始 data.gov.tw 資料集。
- 每日自動更新成功率穩定。
- 當來源失敗或欄位改變時，網站能明確提示，而不是靜默壞掉。

## 8. 設計原則

- **不仿官方目錄感**：官方網站負責發布，本專案負責理解。
- **圖表先有問題意識**：每張圖都要回答一個公共問題。
- **來源永遠可見**：圖表旁必有資料來源卡。
- **限制比結論更重要**：避免用開放資料包裝過度推論。
- **可再製**：處理後資料與程式碼都可追查。
- **中性但不冷漠**：語氣清楚、克制，不做煽動式倡議。

## 9. 風險與取捨

### 主要風險

- 全部資料集格式不一致，很多來源不適合直接視覺化。
- 部分下載連結可能失效、跳轉、壓縮或需要特殊解析。
- 原始資料量可能超過 GitHub Pages 與 repo 適合承載的範圍。
- 中文欄位、民國年、Big5/UTF-8 編碼會增加清洗成本。
- 「議題敘事」容易被誤解為立場倡議，需要方法與限制透明化。

### 建議取捨

- 第一版鏡像全部 metadata，不鏡像全部 raw data。
- 第一版只深做一個能源案例，但資料來源索引要從一開始就做全量。
- 用資料健康度與來源卡建立信任，而不是用大量議題堆疊建立規模感。

## 10. 實作路線圖

### Phase 1：資料來源底座

- 建立 catalog pipeline。
- 定義資料集統一 schema。
- 產生資料集索引 JSON。
- 建立來源異動報告。

### Phase 2：能源議題 MVP

- 完成台電供需資料 ETL。
- 完成近 60 天小時級堆疊面積圖。
- 完成近 12 個月能源結構圖。
- 補上來源卡、方法卡、限制卡。

### Phase 3：議題模板化

- 建立 `issue.yml` 規格。
- 建立議題頁共用元件。
- 建立圖表規格 schema。
- 新增第二個議題，驗證模板不是能源專用。

### Phase 4：搜尋與品質監控

- 加入中文搜尋。
- 加入資料來源健康度儀表板。
- 加入 broken link / schema drift 報告。
- 視資料量評估是否導入 object storage。
